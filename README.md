# BD CRM Analytics

운영 중이던 재구매/추가구매 CRM 대시보드의 **집계 오류를 발견해 지표 기준을 재정의**하고, **실사용자 피드백을 반영해 리포트 구조를 개선**하고, **팀장 검토를 위한 메뉴 매뉴얼을 직접 작성**해 이해관계자와 정렬한 프로젝트입니다.

## 배경 & 문제 인식

기존에 운영 중이던 CRM 대시보드를 점검하던 중, 핵심 지표인 "재구매 회원수"가 실제 정의와 다르게 집계되고 있다는 걸 발견했습니다.

- 서로 다른 상품을 각각 1번씩 산 고객이 "재구매 회원"으로 잘못 잡히고 있었음 (생애 주문 2건 이상 ≠ 재구매)
- "재구매 도래일" 계산에 실제 구매수량이 반영되지 않아, 대용량 구매 고객이 조기에 "이탈"로 오판정되는 구조적 문제

두 문제 모두 운영 지표를 잘못 해석하게 만들 수 있는 사안이라, 원인을 먼저 케이스로 정리한 뒤 기준을 재정의했습니다.

## 기획 및 진행 프로세스

**1. 문제 정의 → 기준 재설계**
"재구매"의 기준을 "동일 상품 2회 이상 구매"로, "재구매 도래일"을 "구매수량을 반영한 예상 주기"로 명확히 재정의했습니다.

**2. 회귀 검증 후 반영**
로직을 바꾸기 전, 수정 전/후 결과를 실제 데이터로 함께 돌려 자동 비교하는 검증 절차를 거친 뒤에만 반영해 — 다른 지표에 의도치 않은 영향이 없는지 확인했습니다.

**3. 이해관계자 정렬 — 메뉴 매뉴얼 작성**
전체 메뉴를 목적별로 정리한 매뉴얼을 만들어, 팀장과 "메뉴와 목적이 실제로 맞는지 / 불필요한 메뉴는 없는지 / 추가로 필요한 기능은 없는지"를 함께 검토할 수 있는 자료로 준비했습니다. 이 과정에서 "캠페인 성과 분석" 메뉴가 실제로는 계산 로직 없이 UI만 존재한다는 것도 함께 찾아내 논의 안건으로 남겼습니다.

**4. 사용성 피드백 반영 — 리포트 재구조화**
"상품별 재구매 통계"와 "상품별 추가구매 통계"가 별도 화면으로 나뉘어 있어 비교하기 불편하다는 실사용 피드백을 받고, 두 리포트를 하나의 "상품별 통합 통계"로 재설계했습니다. 이 과정에서 재구매·추가구매 어디에도 해당하지 않는 "이탈 고객" 지표도 새로 정의해 추가했습니다.

**5. 운영 편의성 개선**
실제 운영진이 느끼던 "응답 속도가 느리다"는 문제를 데이터 기반으로 진단해(로그 계측 → 병목 구간 특정 → 원인 수정) 체감 대기시간을 크게 줄였습니다.

## 활용 방안

- **마케팅팀**: 상품·경과일 조건으로 재구매/추가구매 타겟 고객을 즉시 추출해 캠페인 타겟팅에 활용
- **경영 보고**: 상품별 재구매·추가구매·이탈 현황을 한 화면에서 비교해 상품 포트폴리오 의사결정에 활용
- **팀 협업**: 메뉴 매뉴얼을 기준으로 팀장과 기능 적합성을 논의하고, 캠페인 성과 분석 같은 미구현 기능의 개발 우선순위를 합의하는 데 활용

## 알고 있는 한계 / 다음 논의 과제

- **캠페인 성과 분석**: 현재 UI만 있고 실제 성과 계산 로직은 미구현 — "성과"를 어떻게 정의할지 팀장과 기준 합의가 먼저 필요
- 재구매 판정에 현재 기간 제한이 없음 (예: 1년 전 재구매도 재구매로 집계) — 비즈니스 요구에 따라 판정 기간 옵션 추가 검토 가능

## 사용 기술

Google Apps Script(V8) 기반 서버리스 웹앱, Google Sheets를 데이터베이스로 사용. clasp CLI로 로컬 개발 및 배포.

```bash
npm install -g @google/clasp
clasp login
clasp clone <scriptId>
clasp push
clasp deploy -i <deploymentId>
```
